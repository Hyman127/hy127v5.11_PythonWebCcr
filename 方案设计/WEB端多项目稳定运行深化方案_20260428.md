# Code880 WEB 端多项目稳定运行深化方案

生成时间：2026-04-28

本文基于现有 `WEB端迁移方案.md` 与 `WEB端迁移方案_20260428_架构设计.md` 继续深化，重点回答两个问题：

1. 如何严谨吸收现有 Web 迁移方案中的亮点价值。
2. 如何让同一个浏览器在正常浏览其他网页的同时，稳定打开多个类似本项目的 Web 项目工作台。

## 1. 对现有 `WEB端迁移方案.md` 的亮点吸收

### 1.1 核心价值一：VSCode 不必启动

原方案最关键的判断是：Web 端不需要依赖 VSCode 进程。项目文件树、文件查看、Office/PDF 预览、AI 对话，都可以由本机 Python Web 服务加浏览器前端独立完成。

这个判断非常重要，因为它把系统边界从“操控 VSCode”转为“直接服务项目目录”：

- 文件树由后端读取项目根目录。
- 代码和文本由 Monaco Editor 或文本组件展示。
- Office/PDF 由预览服务转换或前端渲染。
- AI 对话由后端统一代理模型 API。
- 初始化、运行、打包由后端任务系统执行。

因此，Web 端是一个独立的本机工作台，不是 VSCode 的外壳。

### 1.2 核心价值二：浏览器即入口

原方案强调用户双击 `启动Web.bat` 后自动打开浏览器，这是非常符合当前项目“一键安装、一键初始化”理念的。

面向非 IT 用户，应继续坚持：

- 不要求用户打开命令行。
- 不要求用户理解 Python、uv、虚拟环境、PATH。
- 不要求用户先启动 VSCode。
- 只暴露“打开项目”“查看文件”“让 AI 参考”“运行当前文件”等可理解动作。

### 1.3 核心价值三：本地运行与安全边界清晰

原方案提出服务绑定 `localhost` 或 `127.0.0.1`，这符合本项目定位。代码和办公文档默认留在本机，只有 AI 请求在用户选择模型并发起对话时才会联网。

深化后需要继续保持：

- 默认只监听 `127.0.0.1`。
- 默认不开放局域网访问。
- 默认不把项目文件上传到任何服务器。
- AI 上下文只发送用户明确选择或系统按规则选择的片段。
- API Key 只保存在本机后端，不进入前端代码和日志。

### 1.4 核心价值四：三栏工作台布局

原方案的三栏布局非常适合保留：

```text
左侧：项目文件树
中间：文件查看、代码文本、Office/PDF 预览
右侧：AI 对话、上下文、模型选择
底部：运行日志、任务状态、错误输出
```

这套布局的优点是非 IT 用户也容易理解：

- 左边选文件。
- 中间看内容。
- 右边问 AI。
- 底部看运行结果。

多项目方案中，每一个项目工作台都可以沿用这套布局，只是在浏览器中以不同标签页打开。

### 1.5 核心价值五：技术选型轻量、可本机交付

原方案建议：

- 后端：FastAPI + Uvicorn。
- 前端：Vue 3 + Element Plus。
- 编辑器：Monaco Editor。
- PDF：PDF.js。
- Excel：openpyxl 转 JSON 后前端表格展示。
- AI：后端统一封装模型 API。
- 前端打包为静态文件后由 FastAPI 托管。

这套组合适合当前项目，因为它避免了 Electron、Docker、远程服务器等额外复杂度。深化方案建议继续以“一个 Python 本机服务 + 一个浏览器页面”为基本交付形态。

### 1.6 核心价值六：新增 `web/` 目录，不扰动原项目

原方案强调 Web 端放到新增 `web/` 目录，原 `src/`、`__hy127/`、`.vscode/`、初始化脚本保留。这一点必须继承。

原因：

- 现有 VSCode 工作流仍可继续使用。
- Web 端可以逐步实现，不会破坏当前项目。
- 出问题时可退回原流程。
- 便于培训、教学、迭代和灰度试用。

## 2. 多项目稳定运行的总体判断

### 2.1 同一个浏览器可以同时做三类事

同一个浏览器可以同时：

- 正常访问其他网页。
- 打开 Code880 Web 控制台。
- 打开多个项目工作台标签页。

推荐形态如下：

```text
浏览器窗口
  ├─ 标签页 1：普通网页，例如搜索、文档、网盘、邮箱
  ├─ 标签页 2：Code880 Web 控制台
  ├─ 标签页 3：项目 A 工作台
  ├─ 标签页 4：项目 B 工作台
  └─ 标签页 5：项目 C 工作台
```

浏览器只负责显示页面。真正的项目文件读取、任务运行、AI 调用、Office 转换，都由本机后台服务完成。因此只要后台服务设计稳定，多项目并行是可实现的。

### 2.2 不建议每个项目各自占用一个固定端口

简单做法是：

```text
项目 A：http://127.0.0.1:8880
项目 B：http://127.0.0.1:8881
项目 C：http://127.0.0.1:8882
```

这种方式第一版能跑，但长期不推荐，原因是：

- 用户不理解端口。
- 端口容易冲突。
- 浏览器收藏和历史记录混乱。
- 多项目之间缺少统一管理。
- AI 模型配置和登录状态难复用。
- 一个项目崩溃后不容易判断影响范围。

### 2.3 推荐采用“一个 Hub + 多个 Workspace Worker”

推荐架构：

```text
浏览器
  └─ http://127.0.0.1:8800
       ├─ /                       Code880 Web 控制台
       ├─ /w/{workspace_id}        项目工作台页面
       ├─ /api/hub/*               全局项目管理 API
       ├─ /api/workspaces/{id}/*   指定项目 API
       └─ /ws/workspaces/{id}      指定项目实时消息

本机后台
  ├─ Hub 主进程，固定监听 127.0.0.1:8800
  ├─ Project Worker A，服务项目 A
  ├─ Project Worker B，服务项目 B
  └─ Project Worker C，服务项目 C
```

其中：

- Hub 是全局入口和管理器。
- Worker 是每个项目独立的执行服务。
- 浏览器始终访问同一个固定地址 `127.0.0.1:8800`。
- 不同项目用不同 `workspace_id` 区分。
- Worker 可以内部使用动态端口，也可以由 Hub 直接调度，不暴露给用户。

## 3. 推荐架构详解

### 3.1 Hub 主进程职责

Hub 是一个常驻本机服务，建议由 FastAPI 实现。

Hub 负责：

- 提供 Web 前端静态文件。
- 显示项目列表和最近打开项目。
- 注册新项目目录。
- 为每个项目分配 `workspace_id`。
- 启动、停止、重启项目 Worker。
- 维护项目状态、Worker PID、内部端口、最后访问时间。
- 代理浏览器请求到对应 Worker。
- 管理全局 AI 模型配置。
- 管理全局 API Key 加密存储。
- 统一做本地安全认证、Origin 校验、CSRF 防护。
- 保存全局日志和错误报告。

### 3.2 Workspace Worker 职责

每个项目一个 Worker，负责当前项目根目录内的事情。

Worker 负责：

- 读取当前项目文件树。
- 读取和保存当前项目文件。
- 生成当前项目文件预览。
- 管理当前项目 AI 上下文。
- 运行当前项目 Python 文件。
- 执行当前项目初始化、依赖同步、打包等任务。
- 监听当前项目文件变化。
- 写入当前项目日志、备份、审计记录。

Worker 的访问边界必须被限制在项目根目录内。项目 A 的 Worker 不能读取项目 B 的文件。

### 3.3 浏览器前端职责

前端只显示 UI，不直接碰文件系统。

前端负责：

- 显示 Hub 控制台。
- 显示项目工作台。
- 根据 URL 中的 `workspace_id` 访问对应 API。
- 管理当前标签页打开的文件。
- 管理当前标签页选择的上下文文件。
- 展示 AI 流式输出。
- 展示任务日志和运行结果。
- 在浏览器刷新后恢复当前项目 UI 状态。

前端不能保存明文 API Key。模型密钥只能交给后端加密保存。

## 4. 浏览器多标签与普通网页共存设计

### 4.1 多项目用不同标签页承载

推荐每个项目一个浏览器标签页：

```text
http://127.0.0.1:8800/w/code880_temp_v5_11
http://127.0.0.1:8800/w/excel_auto_project
http://127.0.0.1:8800/w/report_tool_project
```

这样用户可以像使用普通网页一样切换项目。浏览器其他普通网页不会被影响。

### 4.2 不依赖浏览器插件

不要要求用户安装 Chrome 插件或 Edge 插件。

原因：

- 插件权限复杂。
- 学员电脑环境差异大。
- 企业电脑可能禁用插件。
- 插件会把本地服务和网页安全边界变复杂。

正确做法是让 Web 工作台作为普通本地网页运行。

### 4.3 每个项目的前端状态必须命名空间隔离

同一个浏览器、同一个域名 `127.0.0.1:8800` 下，多个项目页面会共享 LocalStorage、IndexedDB、Cookie、Service Worker 等浏览器存储。必须主动隔离。

前端存储键必须包含 `workspace_id`：

```text
code880:{workspace_id}:openFiles
code880:{workspace_id}:activeFile
code880:{workspace_id}:panelLayout
code880:{workspace_id}:selectedContextFiles
code880:{workspace_id}:chatDraft
```

不能使用这种全局键：

```text
openFiles
activeFile
selectedContextFiles
```

否则项目 A 的打开文件可能污染项目 B。

### 4.4 同一项目多标签页的协同

用户可能不小心把同一个项目打开两个标签页。此时有两种策略：

策略一：允许多个标签页同时查看，但写入时加锁。

- 适合第一版。
- 多个标签页都可浏览。
- 同一文件编辑保存时，后端检查版本号。
- 如果文件已经被另一个标签页修改，提示用户刷新或对比。

策略二：同一项目只允许一个主编辑标签页。

- 更简单安全。
- 第二个标签页打开时提示“此项目已在另一个标签页打开”。
- 用户可选择“只读打开”或“接管编辑”。

推荐第一版采用“多标签可查看，写入加锁和版本检查”。

### 4.5 普通网页访问本机服务的安全隔离

同一个浏览器中打开普通网页时，恶意网页理论上可能尝试请求 `http://127.0.0.1:8800`。因此不能只依赖“本机地址安全”。

必须做这些保护：

- 后端只允许 `Origin: http://127.0.0.1:8800`。
- 禁止通配 CORS。
- 所有写操作必须携带后端生成的认证 token。
- 所有写操作必须携带自定义请求头，例如 `X-Code880-Token`。
- WebSocket 连接也必须校验 token。
- 启动时生成本机随机 token，普通网页无法读取。
- API Key 不放在 URL，不放在 LocalStorage。
- Cookie 使用 `SameSite=Strict`，写操作再叠加 CSRF token。

这样普通网页可以与项目工作台共存在同一个浏览器中，但不能越权操作本机项目。

## 5. 多项目启动流程

### 5.1 单项目启动

用户在项目 A 双击 `启动Web工作台.bat`：

```text
启动脚本
  -> 检查 Hub 是否已运行
  -> 如果未运行，启动 Hub
  -> 将当前项目路径注册到 Hub
  -> Hub 创建或复用 workspace_id
  -> Hub 启动项目 A Worker
  -> 打开浏览器 http://127.0.0.1:8800/w/{workspace_id}
```

### 5.2 第二个项目启动

用户在项目 B 双击 `启动Web工作台.bat`：

```text
启动脚本
  -> 发现 Hub 已运行
  -> 读取 Hub 端口和认证信息
  -> 将项目 B 路径注册到 Hub
  -> Hub 创建项目 B workspace_id
  -> Hub 启动项目 B Worker
  -> 在同一个默认浏览器中打开新标签页
```

用户感受是：又打开了一个项目网页，不需要理解后台发生了什么。

### 5.3 Hub 发现机制

Hub 启动后写入一个本机状态文件：

```text
%LOCALAPPDATA%\Code880Web\hub_runtime.json
```

示例：

```json
{
  "pid": 12345,
  "port": 8800,
  "base_url": "http://127.0.0.1:8800",
  "started_at": "2026-04-28T17:30:00+08:00"
}
```

启动脚本先读取该文件，再访问：

```text
GET http://127.0.0.1:8800/api/hub/identity
```

如果返回 Code880 Hub 身份，复用现有 Hub。

如果端口被其他软件占用，Hub 自动尝试 `8801`、`8802`，并把实际端口写入 `hub_runtime.json`。用户无需看见端口。

## 6. 多项目数据存储设计

### 6.1 全局数据目录

建议全局数据放在：

```text
%LOCALAPPDATA%\Code880Web\
  hub.db
  hub_runtime.json
  logs\
  keys\
  frontend_cache\
```

全局保存：

- 已注册项目列表。
- 最近打开项目。
- 全局 AI 模型配置。
- 加密后的 API Key 引用。
- Hub 运行日志。
- Worker 运行状态。

### 6.2 项目内数据目录

建议每个项目根目录下新增：

```text
.web-workbench\
  workspace.json
  logs\
  backups\
  preview-cache\
  ai-context-cache\
  task-cache\
```

项目内保存：

- 当前项目工作台配置。
- 文件预览缓存。
- AI 上下文索引。
- 文件修改备份。
- 任务日志。
- 审计记录。

注意：如果希望项目模板保持干净，可以把 `.web-workbench` 加入默认隐藏文件树，但不能无提示删除。

### 6.3 全局与项目数据的边界

全局保存“人和机器的设置”：

- 模型配置。
- API Key。
- 最近打开项目。
- Hub 端口和进程状态。

项目保存“项目自己的运行痕迹”：

- 预览缓存。
- 修改备份。
- AI 对话与当前项目绑定的上下文。
- 任务运行日志。

这样换一台电脑时，项目资料和全局密钥不会混在一起。

## 7. API 路由设计

### 7.1 Hub API

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/hub/identity` | 判断当前端口是否为 Code880 Hub |
| `GET` | `/api/hub/projects` | 获取已注册项目列表 |
| `POST` | `/api/hub/projects` | 注册项目目录 |
| `POST` | `/api/hub/projects/{id}/open` | 打开项目 Worker |
| `POST` | `/api/hub/projects/{id}/close` | 关闭项目 Worker |
| `GET` | `/api/hub/workers` | 查看 Worker 状态 |
| `GET` | `/api/hub/models` | 全局 AI 模型列表 |
| `POST` | `/api/hub/models` | 新增或更新模型配置 |
| `POST` | `/api/hub/models/test` | 测试模型连通性 |

### 7.2 Workspace API

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/workspaces/{id}/info` | 项目状态 |
| `GET` | `/api/workspaces/{id}/files/tree` | 文件树 |
| `GET` | `/api/workspaces/{id}/files/content` | 读取文件 |
| `POST` | `/api/workspaces/{id}/files/save` | 保存文件 |
| `GET` | `/api/workspaces/{id}/preview/pdf` | PDF 预览 |
| `GET` | `/api/workspaces/{id}/preview/office` | Office 预览 |
| `GET` | `/api/workspaces/{id}/preview/excel/data` | Excel 数据 |
| `POST` | `/api/workspaces/{id}/ai/chat` | 项目 AI 对话 |
| `POST` | `/api/workspaces/{id}/tasks/run-python` | 运行指定 Python 文件 |
| `GET` | `/api/workspaces/{id}/tasks/{task_id}` | 查询任务状态 |
| `GET` | `/api/workspaces/{id}/tasks/{task_id}/log` | 查询任务日志 |

### 7.3 WebSocket 路由

```text
/ws/workspaces/{workspace_id}
```

推送内容：

- 文件变更。
- 任务输出。
- AI 流式响应。
- Worker 健康状态。
- 文件锁定状态。
- 预览转换进度。

浏览器断线后自动重连。重连后从后端读取最新任务状态，而不是假设前端内存状态仍可信。

## 8. 稳定运行机制

### 8.1 Worker 隔离

每个项目单独 Worker 的好处：

- 项目 A 的长任务不会堵住项目 B。
- 项目 A 的文件扫描崩溃不会拖垮所有项目。
- 每个项目可以独立重启。
- 资源占用可统计到项目。
- 后续可以对每个项目设置不同 Python 环境。

第一版如果开发成本有限，也可以先做“Hub 单进程多项目”，但架构上应预留 Worker 边界。

### 8.2 心跳与自动恢复

Hub 定期检查 Worker：

```text
每 5 秒：检查 Worker 进程是否存在
每 10 秒：请求 Worker /health
连续 3 次失败：标记异常
如果项目标签页仍活跃：尝试自动重启 Worker
如果重启失败：前端显示恢复按钮和日志入口
```

前端标签页也向 Hub 发送心跳：

```text
每 15 秒：workspace tab heartbeat
用户关闭标签页：停止心跳
项目空闲超过 30 分钟：Worker 可进入休眠
```

### 8.3 任务队列

每个项目维护自己的任务队列。

任务分类：

- 只读任务：读取文件、生成目录摘要、提取文本。
- 预览任务：Office 转换、Excel 分页读取。
- AI 任务：模型调用、上下文构建。
- 写入任务：保存文件、应用补丁、创建文件。
- 命令任务：运行 Python、uv sync、打包。

建议规则：

- 只读任务可以并发。
- 预览任务限制并发数量。
- 写入任务同一时间只能一个。
- 命令任务同一项目默认一次只跑一个。
- AI 流式对话可取消。
- 长任务必须有任务 ID、日志、取消按钮。

### 8.4 文件监听节流

多项目同时打开时，文件监听可能带来大量事件。必须节流。

建议：

- 默认不监听 `.venv`、`.uv-cache`、`__pycache__`、`package/build`、`package/dist`。
- 文件变更事件合并，200 到 500 毫秒内批量推送。
- 大目录只在用户展开时监听或扫描。
- 文件树懒加载，避免一次性递归全部目录。
- 对 OneDrive 同步目录，检测到连续抖动时延迟刷新。

### 8.5 预览缓存

Office/PDF/Excel 预览可能耗时，必须缓存。

缓存键建议：

```text
文件绝对路径 + 文件大小 + 修改时间 + 转换版本
```

缓存目录：

```text
.web-workbench/preview-cache/
```

清理策略：

- 单项目缓存超过指定大小后清理最旧项。
- 原文件修改后自动废弃旧缓存。
- 用户可在设置中清理缓存。

### 8.6 浏览器刷新与恢复

用户刷新项目标签页后，应能恢复：

- 当前打开文件。
- 文件树展开状态。
- 面板布局。
- 当前 AI 会话。
- 正在运行的任务状态。

这些状态不能只放在前端内存里，应同时保存到：

- 前端按 `workspace_id` 命名空间保存轻量 UI 状态。
- 后端保存任务、会话、上下文等关键状态。

### 8.7 资源上限

为保证普通电脑稳定，建议默认上限：

| 资源 | 默认建议 |
|---|---|
| 同时活跃项目 Worker | 3 到 5 个 |
| 单项目并行预览转换 | 1 到 2 个 |
| 单项目命令任务 | 1 个 |
| 单项目 AI 流式对话 | 1 到 2 个 |
| 单个文本文件直接展示上限 | 5MB，可配置 |
| Excel 单次返回行数 | 200 到 1000 行 |
| Worker 空闲休眠时间 | 30 分钟 |

超过上限时不应崩溃，而是提示：

```text
当前同时打开的项目较多，已暂停未使用项目。点击“恢复”可继续。
```

## 9. 多项目 AI 上下文设计

### 9.1 AI 模型配置全局共享

模型配置建议全局保存，多个项目共用。

优点：

- 用户只配置一次 API Key。
- 多项目都能选择同一模型。
- 便于统一测试连通性。

但上下文、对话、文件引用必须按项目隔离。

### 9.2 AI 对话必须带 workspace_id

AI 请求中必须包含当前项目 ID：

```json
{
  "workspace_id": "project_a",
  "conversation_id": "conv_001",
  "message": "请解释这个文件",
  "context_files": ["src/main.py"],
  "model": "deepseek-chat"
}
```

后端根据 `workspace_id` 找到项目根目录，只能读取该项目下的文件。

### 9.3 跨项目引用默认禁止

第一版建议禁止 AI 一次性引用多个项目的文件。

原因：

- 非 IT 用户容易混淆。
- 文件权限边界变复杂。
- AI 修改文件时风险变高。

后期如果需要跨项目比较，可做一个明确功能：

```text
跨项目对比模式
  -> 用户手动选择项目 A 和项目 B
  -> 用户手动选择文件
  -> 只读分析
  -> 默认不允许修改
```

### 9.4 上下文索引按项目独立

每个项目维护独立索引：

```text
.web-workbench/ai-context-cache/
```

索引内容：

- 文件路径。
- 文件类型。
- 文件摘要。
- 分块内容。
- 修改时间。
- 可否进入 AI 上下文。

如果项目文件变更，只更新该项目索引，不影响其他项目。

## 10. 文件写入与并发保护

### 10.1 保存文件需要版本检查

前端读取文件时，后端返回版本信息：

```json
{
  "path": "src/main.py",
  "content": "...",
  "mtime": 1777370000,
  "sha256": "..."
}
```

保存时必须带回：

```json
{
  "path": "src/main.py",
  "content": "...",
  "base_sha256": "..."
}
```

如果文件已被其他进程或标签页修改，后端拒绝直接覆盖，并提示用户对比。

### 10.2 AI 修改必须先备份

AI 应用修改前：

```text
1. 校验路径在项目根目录内。
2. 计算原文件 hash。
3. 写入 .web-workbench/backups/。
4. 展示修改前后对比。
5. 用户确认后写入。
6. 写入审计记录。
```

审计记录包含：

- 时间。
- 项目 ID。
- 文件路径。
- 模型名称。
- 用户指令。
- 修改摘要。
- 备份路径。

### 10.3 多标签写入锁

同一项目的同一文件，只允许一个写入操作进行中。

锁信息：

```text
workspace_id
file_path
owner_tab_id
locked_at
expires_at
```

如果标签页异常关闭，锁过期后自动释放。

## 11. 安全设计补强

### 11.1 本机服务不是天然安全

虽然服务只跑在 `127.0.0.1`，但同一个浏览器中的普通网页仍可能尝试请求本机端口。必须按“本机 Web 应用也可能遭遇 CSRF”的标准设计。

### 11.2 启动认证

Hub 首次启动生成随机 token：

```text
%LOCALAPPDATA%\Code880Web\keys\hub_token
```

启动脚本打开：

```text
http://127.0.0.1:8800/bootstrap?launch_token=一次性token
```

后端校验后设置安全会话。之后所有 API 使用会话和 CSRF token。

### 11.3 请求校验

所有修改类接口必须同时校验：

- 会话是否有效。
- `Origin` 是否为 `http://127.0.0.1:{hub_port}`。
- 是否存在 `X-Code880-CSRF`。
- `workspace_id` 是否存在且归当前会话可见。
- 文件路径是否在项目根目录内。
- 操作是否在权限白名单内。

### 11.4 路径校验

路径校验要考虑 Windows 特性：

- 规范化绝对路径。
- 大小写不敏感比较。
- 拒绝 `..` 路径穿越。
- 拒绝 UNC 路径混入，除非用户明确添加网络盘项目。
- 谨慎处理快捷方式、符号链接、junction/reparse point。
- 禁止默认读取系统敏感目录。

### 11.5 API Key 管理

API Key 不应保存在项目目录，建议保存在全局目录并加密：

```text
%LOCALAPPDATA%\Code880Web\keys\
```

Windows 下优先使用 DPAPI 或 keyring。前端只看到：

```text
模型名：DeepSeek
状态：已配置
Key：sk-****abcd
```

## 12. 用户体验设计

### 12.1 控制台页面

Hub 首页是“项目控制台”：

```text
Code880 Web 控制台
  ├─ 最近打开项目
  ├─ 添加项目
  ├─ 当前运行项目
  ├─ AI 模型配置
  ├─ 全局日志
  └─ 帮助
```

非 IT 用户可以从这里点开项目，不需要记住文件路径。

### 12.2 项目工作台页面

项目页继续沿用原方案三栏布局：

```text
项目 A 工作台
  ├─ 左侧：文件树
  ├─ 中间：文件查看/预览
  ├─ 右侧：AI 助手
  └─ 底部：任务与日志
```

顶部应明确显示当前项目名，避免用户在多个标签页中混淆：

```text
当前项目：Excel自动化练习  |  状态：已准备  |  AI：已连接
```

### 12.3 浏览器标签页标题

浏览器标签标题必须包含项目名：

```text
Excel自动化练习 - Code880
报表生成工具 - Code880
Code880 Web 控制台
```

这样用户在同一个浏览器中切换时更容易辨认。

### 12.4 项目颜色标识

可以给每个项目分配一个温和的颜色标识，用在顶部细线或项目图标上。

注意：颜色只做辅助，不能只靠颜色区分。仍要显示项目名和路径尾部。

### 12.5 关闭项目

关闭浏览器标签页不等于立刻停止 Worker。

推荐规则：

- 标签页关闭后，Worker 保持一段时间。
- 如果 30 分钟无人访问，Worker 休眠。
- 用户可在控制台点击“关闭项目”立即释放资源。
- 如果项目有任务运行，关闭标签页后任务继续，重新打开可查看结果。

## 14. 开发分阶段建议

### 第 1 阶段：单 Hub 单项目

目标：

- 先把原方案跑通。
- 一个 Hub，一个项目。
- 文件树、文本查看、PDF/Excel 基础预览、AI 对话。

验收：

- 不启动 VSCode。
- 双击打开浏览器。
- 当前项目可浏览和问 AI。

### 第 2 阶段：Hub 项目注册表

目标：

- 引入 Hub 控制台。
- 支持添加多个项目路径。
- 支持最近项目列表。

验收：

- 浏览器中能看到多个项目。
- 点击项目打开对应工作台。

### 第 3 阶段：多项目标签页

目标：

- 每个项目拥有独立 `workspace_id`。
- URL 使用 `/w/{workspace_id}`。
- 前端状态按 `workspace_id` 隔离。

验收：

- 同一个浏览器打开项目 A、项目 B。
- 两个项目的文件树、打开文件、AI 对话互不污染。

### 第 4 阶段：Worker 隔离与监督

目标：

- 每个项目独立 Worker。
- Hub 负责启动、心跳、重启。
- Worker 异常不拖垮 Hub。

验收：

- 项目 A Worker 重启时，项目 B 不受影响。
- Hub 控制台能显示每个项目状态。

### 第 5 阶段：稳定性与安全加固

目标：

- 加入认证 token、Origin 校验、CSRF 防护。
- 加入任务队列、文件锁、版本检查。
- 加入缓存清理和空闲休眠。

验收：

- 普通网页无法调用本机写接口。
- 多标签保存同一文件不会误覆盖。
- 同时打开多个项目仍能稳定运行。

## 15. 第一版推荐取舍

第一版不要一次做太重。建议优先实现：

- 单 Hub 固定入口。
- 支持注册多个项目。
- 同一浏览器多标签打开多个项目。
- 前端状态按项目隔离。
- 文件树和文本查看。
- PDF 和 Excel 基础预览。
- AI 模型全局配置。
- AI 对话按项目隔离。
- 每个项目任务日志独立。
- 基础认证和路径安全。

第一版可以暂缓：

- 真正的多进程 Worker。
- 完整 Office 高保真转换。
- 跨项目 AI 对比。
- 多人协作。
- 远程访问。
- 完整在线终端。
- 自动无确认修改文件。

也就是说，产品先实现“一个浏览器能稳定打开多个项目工作台”，再逐步增强“每个项目 Worker 独立隔离”。

## 16. 最终推荐架构结论

推荐最终形态：

```text
一个浏览器
  ├─ 普通网页照常浏览
  ├─ Code880 Web 控制台
  ├─ 项目 A 工作台标签页
  ├─ 项目 B 工作台标签页
  └─ 项目 C 工作台标签页

一个本机 Hub
  ├─ 固定入口
  ├─ 项目注册
  ├─ 模型配置
  ├─ 安全认证
  └─ Worker 管理

多个项目 Worker
  ├─ 项目文件树
  ├─ 文件预览
  ├─ AI 上下文
  ├─ 任务执行
  └─ 日志备份
```

这种架构最符合当前项目的长期目标：

- 对非 IT 用户，入口仍然是一键打开。
- 对多个项目，浏览器用标签页自然承载。
- 对稳定性，项目之间有隔离和恢复机制。
- 对安全性，普通网页不能越权操作本机项目。
- 对未来扩展，AI、预览、运行、打包都能逐步增强。

## 17. 总结

`WEB端迁移方案.md` 的核心亮点是正确的：用本机 FastAPI 服务和浏览器工作台替代 VSCode 显示层，保留项目文件树、文件预览、AI 对话、一键启动和本地安全边界。

在此基础上，多项目稳定运行的关键不是“启动多个 Web 服务抢端口”，而是建立一个稳定的本机 Hub。Hub 作为唯一入口，负责项目注册、模型配置、安全认证和 Worker 管理；每个项目用独立 `workspace_id` 展示，并逐步演进为独立 Worker。

这样，同一个浏览器既可以正常浏览网页，也可以打开多个 Code880 项目工作台。用户看到的是几个普通网页标签页，底层则是受控、隔离、可恢复的本机项目服务体系。
