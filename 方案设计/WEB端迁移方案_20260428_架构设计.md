# Code880 Python 项目工作台迁移到 WEB 端方案

生成时间：2026-04-28

本文基于当前项目目录、`一键安装说明.md`、`重新初始化 V1.24.ps1/.bat`、`.vscode` 配置、`pyproject.toml`、`__hy127` 工具库及初始化日志整理。目标是在不要求用户理解 VSCode、Python 虚拟环境和命令行细节的前提下，把当前“VSCode 打开项目 + 左侧文件树 + 文件查看 + AI 对话/执行”的体验迁移为浏览器中的 Web 工作台。

## 1. 当前项目架构研读结论

### 1.1 项目定位

当前项目不是传统意义上的 Web 应用，而是一个面向 Windows 本机的 Python 开发环境模板：

- `一键安装.exe`：安装 Python、uv、VSCode 便携/本机环境、VSCode Python/debugpy 扩展、pip 镜像、右键菜单等。
- `重新初始化 V1.24.bat/.ps1`：针对当前项目目录初始化 `.venv`，同步依赖，部署 `__hy127`，写入 `.vscode` 配置，运行初始化测试，并尝试启动 VSCode。
- `pyproject.toml`：声明 Python 版本与项目依赖，当前依赖以办公自动化、桌面 GUI、Excel、打包工具为主。
- `src/main.py`：当前业务入口很轻，仅是模板示例。
- `__hy127`：本项目自带的本地工具库，包含 Excel、pandas、日期、日志、Tk/ttkbootstrap 控件、对话框等能力。
- `.vscode`：预置 Python 解释器、F5 运行、当前文件运行、`src/main.py` 运行、保存格式化和推荐扩展。

### 1.2 一键安装说明体现的产品目标

`一键安装说明.md`的核心目标是“零基础用户一键获得 Python + VSCode + uv + 项目模板环境”，重点不是复杂工程开发，而是让非 IT 用户能：

- 不手动配置 Python、PATH、镜像源和 VSCode。
- 解压模板后双击初始化脚本。
- 在 VSCode 中按 F5 运行项目。
- 用 `hy127` 和 pandas/openpyxl/pywin32 等库做办公自动化。

迁移到 Web 端后，产品目标应继续保持“用户只点按钮，不理解底层术语也能上手”。

### 1.3 当前初始化流程可以复用但需要拆分

现有 `重新初始化 V1.24.ps1` 做了很多事情：

- 项目识别：根据文件夹名更新 `pyproject.toml` 项目名。
- 环境重建：删除并重建 `.venv`，保留 `uv.lock`，使用 `.uv-cache`。
- 依赖同步：优先离线 `uv sync --offline`，失败后走镜像，再失败后走 PyPI。
- 工具库部署：运行 `__hy127/init.bat`，把 `__hy127` 部署到 `.venv/Lib/site-packages/hy127`。
- 初始化测试：检查 Python、pandas、arrow、psutil、pyinstaller、pywin32、screeninfo、ttkbootstrap、hy127、中文输出和 GUI。
- VSCode 配置：写入 `.vscode/settings.json`、`launch.json`、`extensions.json`。
- 编辑器启动：找到 VSCode 并打开当前项目。

Web 迁移不应直接把“启动 VSCode”当作必要步骤。更合适的做法是保留环境检测、依赖同步、工具库部署、日志和测试能力，把“打开 VSCode”变成可选动作。

## 2. 关键结论：VSCode 不必启动也能实现

### 2.1 推荐结论

可以不启动 VSCode。Web 端可以独立实现“类似 VSCode 的显示层和交互层”，底层只需要运行一个本机 Web 服务进程。

浏览器负责显示：

- 左侧项目文件树。
- 中间文件文本查看/Office/PDF 预览。
- 右侧 AI 对话和上下文选择。
- 底部运行日志、终端输出、任务状态。

本机后端负责执行：

- 读取项目根目录文件结构。
- 安全读取、写入、预览文件。
- 调用 AI 模型 API。
- 组装文件上下文。
- 运行 Python 脚本、测试、打包等任务。
- 调用现有初始化能力。

### 2.2 为什么不建议把 VSCode 当作必须底座

VSCode 的 UI、扩展市场、编辑器和终端能力很强，但如果让 Web 端依赖“后台运行 VSCode”，会带来这些问题：

- 对非 IT 用户来说，仍然绕不开 VSCode 安装、扩展、窗口状态和后台进程问题。
- Web 前端很难稳定控制 VSCode 内部扩展行为。
- VSCode 扩展的 AI 对话上下文协议不是一个统一、稳定、适合外部 Web 复用的接口。
- 如果使用 code-server 或 OpenVSCode Server，又会引入额外部署、许可、扩展兼容和安全边界问题。

更稳妥的路径是：用 Monaco Editor 实现 VSCode 同款编辑/查看体验，用本机 Python 后端实现项目能力，用 AI Gateway 实现模型注册和工具调用。VSCode 保留为“高级用户可打开”的备用入口。

### 2.3 可选兼容路线

后期可以提供两个入口：

- 普通用户入口：`启动Web.bat`，打开浏览器工作台，不启动 VSCode。
- 高级用户入口：`打开VSCode.bat` 或 Web 内“用 VSCode 打开当前项目”按钮。

这样既满足非 IT 用户，又不牺牲原有 VSCode 工作流。

## 3. 目标体验定义

Web 工作台首屏建议采用三栏布局：

```text
┌────────────────────────────────────────────────────────────────────┐
│ 顶栏：项目名称 / 初始化状态 / 模型选择 / 运行按钮 / 设置            │
├───────────────┬──────────────────────────────────┬─────────────────┤
│ 项目文件       │ 文件查看 / 预览 / 编辑             │ AI 助手          │
│               │                                  │                 │
│ 文件夹树       │ 代码和文本：文本方式展示            │ 当前上下文文件     │
│ 文件搜索       │ Word/PPT/Excel/PDF：格式化预览     │ 对话             │
│ 常用文件       │ 图片：图片预览                     │ 执行计划/结果     │
├───────────────┴──────────────────────────────────┴─────────────────┤
│ 底部：任务日志 / Python 输出 / 初始化结果 / 文件变更记录            │
└────────────────────────────────────────────────────────────────────┘
```

用户打开后应该能直接看到：

- 当前项目是否已初始化。
- 左侧项目文件夹层级。
- 点击文件后的内容。
- 哪些文件已被加入 AI 上下文。
- AI 做了什么、改了哪些文件、运行结果是什么。

## 4. 推荐总体架构

### 4.1 架构分层

```text
浏览器 Web UI
  ├─ 文件树 Explorer
  ├─ 文件查看/预览 Viewer
  ├─ Monaco 文本查看/编辑器
  ├─ AI 对话面板
  ├─ 任务日志和运行结果
  └─ 设置/模型管理

本机 Web 后端 FastAPI
  ├─ Project Service：项目根目录、初始化状态、环境检测
  ├─ File Service：文件树、读取、保存、搜索、diff、备份
  ├─ Preview Service：Office/PDF/图片/文本预览
  ├─ AI Gateway：模型注册、密钥管理、对话代理、流式输出
  ├─ Context Engine：文件上下文、目录摘要、索引、token 控制
  ├─ Task Runner：运行 Python、uv、测试、打包、初始化
  └─ Security Guard：路径限制、命令限制、敏感文件保护、审计日志

当前项目运行环境
  ├─ .venv/Scripts/python.exe
  ├─ uv / uv.lock / .uv-cache
  ├─ src/
  ├─ __hy127/
  ├─ pyproject.toml
  └─ .vscode/ 作为兼容配置保留
```

### 4.2 技术选型建议

后端：

- Python 3.12，延续当前环境。
- FastAPI + Uvicorn：提供本机 Web API 和 WebSocket。
- Pydantic：接口数据校验。
- watchdog：监听文件变化，刷新文件树。
- SQLite：保存模型配置、对话记录、文件索引、审计日志。
- subprocess/asyncio：运行 Python、uv、测试和打包任务。
- Windows DPAPI 或 keyring：本机加密保存 API Key。

前端：

- React + TypeScript + Vite。
- Monaco Editor：实现 VSCode 风格文本查看/编辑。
- PDF.js：PDF 预览。
- SheetJS 或后端 openpyxl/pandas：Excel 表格预览。
- office 文件优先由 LibreOffice headless 转 PDF/HTML 预览；如果不内置 LibreOffice，则提供轻量解析预览。
- xterm.js：后期可做终端式运行输出。

打包/启动：

- 开发期：`启动Web.bat` 调用 `.venv/Scripts/python.exe -m uvicorn ...`。
- 交付期：可用 PyInstaller 打包为 `Web工作台.exe`，首次启动时自动打开浏览器。
- 不建议第一阶段引入 Docker，目标用户是 Windows 本机零基础用户。

## 5. 功能模块设计

### 5.1 项目文件树

目标：Web 端左侧拥有类似 VSCode 的项目根目录文件夹层级展示。

功能：

- 显示项目根目录下的文件夹和文件。
- 支持展开/收起、刷新、搜索、按文件类型筛选。
- 支持文件图标、大小、修改时间、是否隐藏。
- 支持右键菜单：复制路径、在资源管理器中打开、加入 AI 上下文、从上下文移除。
- 支持拖拽上传文件到指定目录，后期再开放。

默认隐藏建议：

- `.venv`
- `.uv-cache`
- `.git`
- `__pycache__`
- `package/build`
- `package/dist`
- `*.pyc`
- 大型二进制缓存文件

注意：隐藏不等于不可访问，应该在设置中提供“显示系统/缓存目录”开关，避免高级用户找不到文件。

### 5.2 文件查看与预览

目标：点击左侧具体文件时，默认以用户能理解的方式展示。

文件类型策略：

| 类型 | 默认展示 | 说明 |
|---|---|---|
| `.py/.md/.txt/.json/.toml/.ps1/.bat/.csv/.log` | 文本查看 | 代码类也按文本展示，可使用 Monaco 提供行号和搜索 |
| `.docx/.doc` | Word 预览 | 优先转 PDF/HTML 展示，保留分页和格式；同时提供“提取文本” |
| `.pptx/.ppt` | PPT 预览 | 优先转 PDF/图片分页展示 |
| `.xlsx/.xls` | 表格预览 | 工作表 Tab、冻结表头、分页加载、单元格文本展示 |
| `.pdf` | PDF 预览 | PDF.js 翻页、缩放、搜索 |
| 图片 | 图片预览 | 缩放、适应窗口、下载/打开目录 |
| 其他二进制 | 基本信息 | 显示大小、路径、修改时间，提示无法直接预览 |

重要说明：

- 当前一键安装包主要安装 Python、uv、VSCode 和 Python 依赖，并未完整包含 Web Office 预览链路。
- 如果要求 Word/PPT/Excel/PDF 在 Web 中“按相应格式展示”，需要在 Web 版本安装包中新增预览依赖。
- 推荐第一版使用“后端转换 + 浏览器预览”的方式，避免让用户安装 Office。

Office 预览推荐策略：

- PDF：直接用 PDF.js。
- Word/PPT：优先内置或检测 LibreOffice 便携版，使用 headless 转 PDF，再用 PDF.js 展示。
- Excel：优先用 openpyxl/pandas 读取结构化数据，Web 端以表格网格展示；复杂格式可提供“转 PDF 预览”作为增强。
- 加密或损坏文件：给出明确错误信息，不直接崩溃。

### 5.3 AI 模型注册

目标：通过 API 注册 AI 模型，拥有类似 VSCode AI 对话模型的能力。

模型配置字段建议：

```json
{
  "name": "工作模型",
  "provider": "openai_compatible",
  "base_url": "https://api.example.com/v1",
  "api_key_ref": "local_encrypted_key_id",
  "model": "model-name",
  "supports_stream": true,
  "supports_tools": true,
  "max_context_tokens": 128000
}
```

设计要点：

- 支持 OpenAI-compatible API，便于接入多种模型服务。
- API Key 只保存在本机后端，不传给前端明文。
- 前端只能看到模型名称、服务商、是否可用，不能看到完整密钥。
- 支持模型连通性测试。
- 支持默认模型、备用模型、只读问答模型、执行型模型分开配置。

### 5.4 AI 上下文机制

目标：AI 能根据左侧文件夹层级里的具体文件作为上下文。

上下文来源：

- 用户手动勾选文件。
- 当前打开文件。
- 当前选中的目录摘要。
- 最近修改文件。
- 搜索命中的文件片段。
- 项目结构摘要。
- 初始化日志、运行日志、错误堆栈。

上下文控制：

- 对文本文件按大小和 token 分块。
- 大文件只取相关片段，避免整文件塞入模型。
- 对 Office/PDF 文件先提取文本，再按页/表/段落引用。
- 对 Excel 文件可按工作表、列名、前 N 行、命中行摘要注入。
- 对二进制文件只注入元数据。
- 对 `.env`、密钥、证书、token 文件默认不注入。

AI 回答需要附带来源：

```text
参考文件：
- src/main.py
- pyproject.toml
- 一键安装说明.md
```

当 AI 准备修改文件时，建议先显示变更摘要和 diff，让用户点“应用修改”。面对非 IT 用户，也可以提供“备份并应用”按钮。

### 5.5 AI 执行能力

“类似 VSCode 的 AI 模型执行”不应只做聊天。后端需要给 AI 暴露受控工具：

- 读取文件。
- 搜索文件。
- 生成补丁。
- 应用补丁。
- 运行当前 Python 文件。
- 运行 `src/main.py`。
- 运行初始化检查。
- 运行打包命令。
- 查看日志。
- 创建文件备份。

执行边界：

- AI 只能访问当前项目根目录。
- 文件写入必须经过后端路径校验。
- 高风险命令需要前端确认。
- 删除、移动、批量覆盖要有备份和审计记录。
- 默认禁止读取系统目录和用户其他项目。

### 5.6 初始化与项目状态

Web 端应提供“项目初始化”页面或顶部状态条：

- 当前 Python 是否存在。
- `.venv` 是否存在。
- `uv.lock` 是否存在。
- 依赖是否同步。
- `hy127` 是否部署。
- 初始化测试是否通过。
- VSCode 配置是否存在。

推荐新增一个 Web 专用初始化流程：

```text
启动Web.bat
  -> 检查 .venv/Scripts/python.exe
  -> 若不存在，提示执行初始化
  -> 运行 Web 后端
  -> 自动打开浏览器
  -> Web 内展示初始化状态和日志
```

现有 `重新初始化 V1.24.ps1` 可保留，但建议后续拆出公共逻辑或新增参数：

- `-SkipVsCodeLaunch`：初始化但不启动 VSCode。
- `-WebMode`：输出结构化 JSON 状态，便于 Web UI 读取。
- `-NoGuiTest`：服务器环境或无桌面时跳过 Tk GUI 测试。

## 8. 非 IT 用户体验细节

### 8.1 文案风格

不要使用“虚拟环境、解释器、token、diff、subprocess”等术语作为主文案。可以这样替换：

- “初始化环境”改为“准备项目”。
- “依赖同步”改为“安装项目需要的组件”。
- “运行脚本”改为“运行当前文件”。
- “上下文”改为“让 AI 参考这些文件”。
- “diff”改为“修改前后对比”。

### 8.2 AI 操作流程

推荐流程：

1. 用户点击一个文件。
2. 用户点“让 AI 参考此文件”。
3. 用户输入问题或选择快捷操作，如“解释这个文件”“帮我修改”“运行并看错误”。
4. AI 给出回答或修改建议。
5. 如果要改文件，Web 显示修改前后对比。
6. 用户点“备份并应用”。
7. Web 自动运行检查，并显示结果。

### 8.3 防误操作

默认提供这些保护：

- 修改前自动备份到 `.web-workbench/backups/`。
- AI 执行命令前显示命令含义。
- 删除文件需要二次确认。
- `.env`、密钥、证书默认遮蔽。
- 日志可一键复制给老师或技术支持。

## 9. 安全与合规边界

必须落实：

- 后端只监听 `127.0.0.1`，默认不允许局域网访问。
- 所有文件路径都必须经过项目根目录校验，防止路径穿越。
- API Key 本机加密保存，不写入普通日志。
- 对话日志中默认不记录完整密钥、密码、token。
- AI 不能直接执行任意 PowerShell 或 CMD，只能调用白名单任务。
- 高风险文件如 `.env`、证书、浏览器缓存、系统目录不进入 AI 上下文。
- Office/PDF 转换要使用临时目录，并在转换后清理。
- 所有 AI 修改记录写入审计日志，便于回滚。

## 12. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| Office 格式复杂 | Word/PPT 排版转换可能不完全一致 | 第一版优先转 PDF；失败时给出提取文本 |
| AI 上下文过大 | 项目文件多、日志长 | 文件分块、目录摘要、用户选择、token 限制 |
| AI 误改文件 | 非 IT 用户难以判断 | 自动备份、diff、确认按钮、撤销 |
| 命令执行危险 | AI 可能建议删除或运行危险命令 | 命令白名单和风险分级 |
| API Key 泄露 | 密钥写入日志或前端 | 后端加密保存，日志脱敏 |
| VSCode 依赖混乱 | 用户误以为必须开 VSCode | Web 入口默认独立运行，VSCode 作为可选 |
| 预览大文件卡顿 | Excel/PDF/日志可能很大 | 分页、懒加载、大小限制 |
| OneDrive 路径同步冲突 | 当前项目在 OneDrive 下，文件可能被占用 | 写入前检测锁定，备份目录避开频繁改名 |

## 13. 第一版最小可行产品范围

建议第一版先做“可用、稳定、不会误伤文件”的版本：

必须有：

- Web 启动入口。
- 项目文件树。
- 文本查看。
- PDF 查看。
- Excel 基础表格查看。
- AI 模型注册。
- AI 对话。
- 手动选择文件作为 AI 参考。
- 运行 `src/main.py`。
- 任务日志。

暂缓：

- 完整代码补全。
- 完整调试器。
- 多人协作。
- 远程公网访问。
- 完全复刻 VSCode 扩展生态。
- AI 自动无确认修改文件。

## 14. 推荐下一步

建议下一步不要直接大改原项目，而是新增一个独立 `web/` 骨架，并保持原 VSCode 流程可用：

1. 新增 `web/backend` 与 `web/frontend`。
2. 新增 `启动Web工作台.bat`。
3. 后端先只读项目根目录并返回文件树。
4. 前端先实现左侧文件树和中间文本查看。
5. 再接入 AI 模型注册和上下文。
6. 最后逐步开放文件写入和命令执行。

最终形成两个并行入口：

- 面向普通用户：双击 `启动Web工作台.bat`，浏览器中完成查看、问答、运行。
- 面向开发用户：继续使用 `重新初始化 V1.24.bat` 和 VSCode。

## 15. 总结

本项目完全可以迁移为 Web 端工作台，而且不需要 VSCode 启动。更准确地说，Web 端不应“套住 VSCode”，而应把当前项目的运行环境、初始化脚本、文件系统、Python 执行能力和 AI 能力封装到本机后端，再用浏览器提供一个更友好的工作界面。

这样做能保留现有一键安装和 Python 模板的优势，同时把用户从 VSCode 的概念负担中解放出来。对于不懂 IT 的用户，最终感受会是：双击打开一个网页，看文件、问 AI、点按钮运行和修复，而不是学习编辑器、终端和环境配置。
